# TTS / Perception Agent · 体验层级分析

> **发言时间**: SESSION-001, 第七席
> **视角**: 感知层不是表面的装饰，是架构的最后一公里
> **核心问题**: AI 辩论的"感觉"是装饰还是核心？

---

## 1. 感知层不是 UI，是接口

传统架构把"感知"看作前端的事情——UI 设计师画个图，前端工程师写个页面。但在我们的语境下，**感知层是 L3 系统层到 L4 产品层的正式接口**。

```
L3 系统层产出:
  - 辩论文本 (episode)
  - 评分数据 (scores)
  - 教练策略 (strategy)
  - 讲茶大堂评论 (teahouse)

L4 产品层展示:
  - 对话流 (需要 TTS 合成语音)
  - 雷达图 (需要评分数据)
  - 螺线谱 (需要音高数据)
  - 比赛回放 (需要完整 episode)

感知层 = 把 L3 的结构化数据翻译成人类感官可接收的信号
```

这个翻译不是一种而是多种：

```
结构化数据 → 视觉信号 (雷达图 / 螺线谱 / 对话界面)
结构化数据 → 听觉信号 (TTS 语音 / MIDI 背景音乐)
结构化数据 → 叙事信号 (赛后报告 / 精彩集锦 / 关键片段高亮)
```

每一种翻译都是一个独立的 L2 模块，各自有输入输出契约。如果我只给 L3 数据不给感知层，L4 产品层就是空的。

## 2. TTS 的核心架构选择

TTS 不是"把文字念出来"，在辩论场景中 TTS 是**角色塑造**的关键手段。

```
TTS 的两个极端:

极端 A: 中性 TTS
  所有角色用同一个声音, 只是念对不同名字
  优点: 实现简单, 成本低
  缺点: 观众分不清谁在说话, 体验单调

极端 B: 角色 TTS
  每个"八仙"有独立声纹
  优点: 沉浸感强, 角色辨识度高
  缺点: 需要 8 个声纹模型, 部署成本高, 声纹相似度可能串
```

**推荐路径: 两阶段 TTS 策略**

```
Phase 1 (V2.0, MVP):
  单声纹 TTS + 文本前缀:
  "[吕洞宾] 天地不仁, 以万物为刍狗"
  "[何仙姑] 但天地也有慈悲的一面"
  声纹: 1 个中性男声
  实现: edge-tts 原生支持, 零配置

Phase 2 (V3.0, polished):
  多声纹 TTS:
  8 个 unique speaker profiles
  声纹: 通过 Omni-TTS seed + ref_audio 控制
  实现: 独立的 TTS 微服务, 支持语音切换
```

关键架构决策: TTS 是**同步**还是**异步**？

```
同步 TTS:
  辩论每轮结束 → 立即生成语音 → 拼接到对话流
  优点: 实时, 观众可以边读边听
  缺点: 阻塞辩论流程 (TTS 时间 + 辩论时间)
  适用: 直播场景

异步 TTS:
  辩论结束后 → 生成完整 episode → 批量 TTS
  优点: 不阻塞, 可以并发
  缺点: 没有实时感
  适用: 赛后回放 / 播客生成
```

**推荐**: 异步 TTS（辩论时不 TTS，赛后批量生成）。这不是功能取舍，是架构决策——同步 TTS 让辩论引擎的复杂度翻倍（需要处理 TTS 队列和状态同步），而异步 TTS 只需要一个独立的后处理管道。

## 3. 螺线谱的架构层级

螺线谱是视觉化工具中最「鲲鹏志」的一个组件，也是架构上被最低估的。

```
螺线谱不是:
  - 装饰性的图表
  - 比赛完才生成的"事后好看"
  - 前端工程师自己就能做的东西

螺线谱是:
  - 辩论过程中的实时反馈
  - 音高数据的可视化表达
  - 盲测特征的视觉指纹 (每队的螺线轨迹不同)
```

螺线谱在架构中的位置:

```
数据源 (L3.2):
  辩论文本 → 音高/情绪分析 → 音频特征序列
  (从文本中提取节奏感: 句子长度、标点频率、情绪曲线)
  
映射层 (L2):
  音频特征 → 螺旋坐标系
  时间轴 → 螺旋角度
  音高 → 螺旋半径
  情绪强度 → 颜色饱和度

渲染层 (L1):
  螺旋坐标 → Canvas 2D / WebGL
  (渲染在当前 MBP 上, 不做 SSR)
```

**关键架构选择**: 螺线谱的计算应该在服务端还是客户端？

```
方案 A: 服务端计算
  辩论引擎输出 episode → 服务端解析文本 → 生成螺旋数据
  前端只负责渲染
  优点: 计算负载在服务端, 前端薄
  缺点: 需要额外的 L2 模块 (text_to_spiral)
  Token 成本: 每次解析约 5K tokens

方案 B: 客户端计算
  辩论引擎输出 episode → 前端拿到全文 → 前端解析 + 渲染
  优点: 不需要额外模块
  缺点: 前端重, 初始加载慢
  Token 成本: 0

方案 C: 混合 (推荐)
  服务端预计算关键特征 (情绪曲线、关键转折点)
  客户端实时渲染 (把特征向量画成螺旋)
  优点: 分工合理
  缺点: 需要定义"特征"和"渲染"的接口
  Token 成本: ~2K tokens/场 (比全量少 60%)
```

## 4. 三套叙事的时间层级

辩论结束后，同样的数据可以产三种不同的叙事，各自有不同的"最佳食用时间"：

```
叙事 A: 即时战报 (比赛结束后 5 分钟)
  格式: 文字短文 + 胜负核心数据
  长度: 300-500 字
  受众: 泛观众, 社交分享
  生成方式: LLM 摘要 (烧 ~5K tokens/场)
  发布: 比赛结束后自动生成

叙事 B: 深度分析 (比赛结束后 2 小时)
  格式: 长文分析 + 雷达图 + 关键片段
  长度: 1500-3000 字
  受众: 技术观众, 参赛厂商
  生成方式: LLM 分析 (烧 ~20K tokens/场)
  发布: 有人工审核

叙事 C: 播客 (比赛结束后 24 小时)
  格式: 音频节目 + 双人对话
  长度: 15-30 分钟
  受众: 通勤观众, 长尾消费
  生成方式: TTS + 播客脚本 (烧 ~50K tokens/场)
  发布: 有编辑审核
```

这三个叙事的时间层级不是技术限制，是**节奏设计**——给观众一个"逐渐深入"的体验，而不是一次把全部数据砸过去。

## 5. 情感渲染曲线

辩论有情感曲线。开场 → 冲突 → 高潮 → 尾声。这个曲线应该在辩论开始前就规划好，而不是赛后回顾时才看到。

```
情感曲线可以指导:
  - MIDI 背景音乐的强度变化
  - 螺线谱的颜色饱和度
  - 讲茶大堂评论的调侃程度 (开场轻松, 高潮时犀利)
  - TTS 语速 (高潮时加快, 尾声时放缓)
```

**架构影响**: 情感曲线需要成为 L3.3 教练层输出的一个组成部分。

```
Coach 的 strategy_doc 中新增:
  "emotional_arc": {
    "opening": "neutral",
    "round_1_2": "building_tension",
    "round_3_4": "climax", 
    "round_5_6": "resolution",
    "round_7_8": "reflection"
  }
```

这 5 个情感阶段影响感知层所有组件的参数。

## 6. 感知层的"无障模式"

最后，感知层有一个经常被忽视的设计维度：**当感知手段不可用时的降级。**

```
场景:
  用户在地铁上 (不能听 TTS, 只能看文字)
  用户是视障人士 (只能听 TTS, 不能看雷达图)
  用户网速慢 (螺线谱加载不出来)
  用户 Safari 浏览器 (WebGL 不支持)
```

每个降级场景对应一种"感知降级路径":

```
不可听 → 文字流 + 速度不变字幕
不可看 → TTS 全量朗读 + 比赛总结
网速慢 → 先出文字, 再出螺线谱
设备差 → 2D Canvas 替代 WebGL
```

**架构要求**: 感知层的每个组件都应该有 `fallback_renderer`，在主渲染路径不可用时自动切换。

---

## 结论

感知层不是 UI 设计的子集，是 L3 到 L4 的正式接口层。三个架构要求：

1. **TTS 异步化** — 不阻塞辩论引擎，赛后批量生成
2. **螺线谱服务端预计算** — 服务端做特征提取，客户端只负责渲染
3. **情感曲线作为教练策略输出** — 从辩论开始就规划情感路线
4. **感知降级路径** — 每个感知组件都有 fallback 渲染器
