# Security Agent · 身份与信任层级分析

> **发言时间**: SESSION-001, 第一席
> **视角**: 全局信任模型
> **核心问题**: 盲测体系的信任基座在哪里崩塌？

---

## 1. 信任层级本质

鲲鹏志的立身之本是盲测。没有盲测，就没有"客观的模型能力对比"，就没有厂商愿意来。

但信任不是二进制的（可信/不可信），信任是**层级递进的**：

```
L0 信任: 你声称你是盲测 → 我相信
L1 信任: 我看到盲测机制 → 我接受
L2 信任: 我验证了盲测结果 → 我认可
L3 信任: 即使结果对我不利, 我也相信机制公正 → 我持续参与
```

我们现在的位置: L0 到 L1 的过渡期（Issues 里在讨论盲测，但还没有公开验证过的盲测轮次）。

**要到 L3, 需要回答两个问题:**
1. 如果一场比赛的盲测结果和厂商自己内部评测矛盾，厂商信谁？
2. 如果一个厂商拿到了对自己不利的证据，它能做什么？

## 2. 信任基座的六个崩塌点

### 崩塌点 1: 盲测映射表的存储层级

盲测映射表（哪个 blind_name 对应哪个模型）是这个系统里**最敏感的数据**。

```
当前设计风险:
blind_map 存在于 MatchPlan 对象中
→ MatchPlan 被 Tournament Orchestrator 持有
→ Orchestrator 在 L3.1 层
→ 如果 L3.1 的任何日志打印了 MatchPlan → 盲测泄露

建议层级隔离:
blind_map 加密存储, 只有以下条件可解密:
├─ 比赛中: 不可解密 (连 Orchestrator 也不持有明文)
├─ 赛后评分完成: L4 产品层触发解密
└─ 审计: 只有 L5 战略层有权限主动查询
```

这里有个架构选择：**Orchestrator 是持盲测映射还是不持？**

如果持 → 简单但风险高（日志泄露、debug 泄露）
如果不持 → 安全但复杂（需要额外的解密调度层）

建议：L3.1 编排层**不持明文的 blind_map**。盲测映射在 Tournament 初始化时生成→加密→存入 L3.6 存储层。比赛中 Orchestrator 只持有 `blind_name`，不知道真身。赛后 L4 触发解密→评分。

### 崩塌点 2: 模型指纹的被动泄露

即使 blind_map 是安全的，模型在被调用时也会留下指纹：

```
指纹类型                    暴露层级
──────────────────────────────────────────────
回复速度 (TTFT, TPS)        L1 工具层 (API 响应)
输出格式习惯 (Markdown 风格)  L2 辩论模块 (prompt 模板)
倾向性价值判断                L3.2 辩论引擎 (内容层面)
Token 分布模式                L1 工具层 (tokenizer 差异)
```

这些指纹叠加，就像 DNA——单一维度可能不明显，但 8 轮辩论下来，所有维度的叠加指纹可以高精度定位模型身份。

**架构应对**: L2 辩论模块需要增加一层**输出归一化**:

```
原始输出 → 格式归一化 (去 Markdown 差异)
         → 速度归一化 (延迟抖动填充)
         → 标点归一化 (统一到目标风格)
         → 匿名化输出 (传递给 Orchestrator)
```

这层不是安全模块，是**辩论引擎的组成部分**。它保障的不是数据安全，是比赛公平。

### 崩塌点 3: 赛后解密的时间窗口

盲测最有价值的时刻不是比赛中，而是**评分全部入库后、解密公布的那一瞬间**。

如果解密后厂商发现:"不对，我在 X 维度应该是第一的，这个评分有问题"——这时候信任基座已经开始动摇。

**架构建议**: 赛后解密不是一次性的。分三阶段:

```
Phase 1 (比赛结束 +0h):  公布盲测 ID 和评分的对应关系
Phase 2 (比赛结束 +24h): 公布盲测 ID 和模型真身的对应关系
Phase 3 (比赛结束 +72h): 发布完整审计日志 (所有 API 调用的输入输出)
```

这给厂商 24 小时"消化"评分结果，再看身份匹配——避免"评分结果+模型身份"同时冲击造成的争议放大。

### 崩塌点 4: 历史数据的反推

假设 V2.1 运行了 20 场比赛，某个厂商参加了其中 5 场。

```
该厂商可以:
1. 记录自己模型在所有场次的所有发言
2. 赛后收到盲测解密
3. 把 "自己模型的发言特征" 做特征工程
4. 去反推之前未解密场次中 "哪些发言风格像自己" → 推断其他盲测场的模型分配
```

这是**数据级副作用**，不是代码 bug。需要架构层面解决：

**措施**: 同一模型在连续两场锦标赛中的 blind_name 必须不同。同一厂商如果带多个模型参赛，每个模型的 blind_name 在赛季内不重复使用。

### 崩塌点 5: 作弊的经济学

任何安全体系都可以被突破，关键是突破的成本和收益。

```
作弊收益:
├─ 赢得一场 4v4 → 厂商可以发 PR 稿: "我们的模型在盲测中战胜了 GPT-4o"
└─ 收益 ≈ 百万级 PR 价值

作弊成本:
├─ 识别对方模型: 需要跑 8 轮辩论采集指纹
├─ 让本方模型针对性攻击: 需要实时注入提示攻击
└─ 成本 ≈ 几千 token 的推理调用 + 1 个工程师的半天工时
```

**经济等式**: 收益远大于成本。如果不做防御，作弊是理性选择。

**架构回应的核心不是技术，是声望**：
- 如果作弊被逮到 → 公开除名 → 声望损失 >> 一次 PR 收益
- 但 "被逮到的概率" 取决于 L2 输出归一化 + L3 评分层指纹检测
- 检测率 > 90% 时 → 作弊成为负期望值行为

### 崩塌点 6: 评审员的信任

BYOK 评分员的引入带来了一个有趣的信任悖论：

```
场景: 某厂商 BYOK 给所有模型打的分数都集中在 3.0±0.2, 而官方评分器的分数范围是 1.5-4.8

这意味着什么?
├─ 可能性 A: 该 BYOK 评分员不负责任, 全打中分
├─ 可能性 B: 该厂商的 API Key 是免费层, 上下文不够, 评分不准
└─ 可能性 C: 该评分员故意减小方差, 降低自家评分的影响力 (反向操纵)
```

所有三种可能性都是系统问题，不是评分员的问题。架构需要：

1. BYOK 评分质量指标: 每个 BYOK 评分附带一个 `confidence` 分数（基于评分分布和被评模型的多样性）
2. 评分员信誉系统: 每个 BYOK 账号有 `reputation` 分，偏离共识太多 → reputation 衰减
3. 加权投票: 评分权重 = BYOK 评分 × reputation × confidence

## 3. 跨层级信任链路

```
L5 战略层 ── 承诺: "我们做的是公正盲测"
     ↓ 信任需要往下传递
L4 产品层 ── 展示: "这是你的模型的真实表现"
     ↓ 数据需要被信任
L3 系统层 ── 执行: "盲测映射已加密, 评分已存证"
     ↓ 机制需要可审计
L2 模块层 ── 隔离: "输出已归一化, 指纹已掩盖"
     ↓ 实现需要可验证
L1 工具层 ── 运行: "API 调用已记录, 日志已封存"
```

这条链上**任何一环的信任断裂都会导致全链失效**。
如果 L1 的日志被人为篡改，L5 的承诺就是空话。

**架构落地**: 链上哈希链 (简单版本)
- 每场比赛结束后，对 episode、scores、blind_map 做一次 SHA256
- 将哈希值推送到一个公开可见的地方（GitHub Release / 飞书公开文档）
- 这样任何人都能验证: "这场比赛的数据没有被篡改过"

## 4. 给系统的威胁模型评级

| 威胁 | 发生概率 | 影响程度 | 优先级 |
|------|---------|---------|-------|
| 盲测映射意外在日志泄露 | 中 | 致命 | P0 |
| 输出指纹暴露模型身份 | 高 | 高 | P0 |
| BYOK 评分被批量操纵 | 中 | 高 | P1 |
| 赛后解密争议 | 高 | 中 | P1 |
| 历史数据反推 | 低 | 中 | P2 |
| 哈希链被篡改 | 极低 | 致命 | P2 (但处理成本低, 值得做) |

---

## 结论

**盲测不是安全功能，是产品功能。** 它的可信度决定了大厂是否愿意把模型交给我们。我从 6 个方向系统性地审视了信任基座的崩塌点，给出了对应的架构层级应对方案。核心建议：

1. blind_map 加密隔离 — L3.1 不持明文
2. 输出归一化层 — L2 辩论模块增加指纹掩盖
3. 三阶段解密 — L4 产品层控制解密节奏
4. 哈希链存证 — 每场比赛的结果可公开验证
