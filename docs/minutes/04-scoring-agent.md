# Scoring Agent · 公允性层级分析

> **发言时间**: SESSION-001, 第四席
> **视角**: 评分层的权力与制衡
> **核心问题**: 当评分结果和直觉不符时，谁有最终解释权？

---

## 1. 评分不是测量，是权力

这不是一个哲学感慨，是架构层面的现实。

当系统输出 "模型 A 在反驳精准度上 4.2 分，模型 B 3.1 分" 时，这个数字决定了：
- 模型 A 的团队可以拿这个数字去申请更多训练资源
- 模型 B 的团队需要向老板解释为什么比对手差了 35%
- 外部观众用这个数字判断"哪个模型更聪明"

**评分 = 分配声望的权力。** 谁掌握评分权，谁就掌握话语权。

当前架构中评分权分配:

```
官方评分器 (LLM)    → 60% 的评分权重  ← 系统默认
BYOK 评分员         → 20% 的评分权重  ← 用户自带 token
教练遵从度评分       → 10% 的评分权重  ← 系统副产物
观众投票            → 10% 的评分权重  ← V3.0 规划中
```

问题是：**这个权重分配为什么是这样而不是其他比例？** 如果没有任何依据地分配权力，那就不是架构，是专断。

**架构要求**: L3.4 评分层的权重需要一套明确的「权力分配逻辑」：

```
权重的确定因素:
  1. 数据量: 评分样本越多 → 权重越高 (但 BYOK 评分量受限于用户)
  2. 历史一致性: 过去与共识偏离越小 → 权重越高
  3. 维度覆盖: 对所有维度的评分离散度 → 
     离散度高说明认真区分了维度 → 权重高
     离散度低 (全打 3 分) → 权重低
  4. 时间衰减: 距离评分时间越久 → 权重越低 (因为用户可能忘细节了)
```

这个逻辑要在 L4 产品层公开（透明化），让厂商知道权重是怎么算的。

## 2. 评分维度本身是最强的偏见

六维雷达选的六个维度本身就隐含了价值判断：

```
现有六维:
  战术执行度: "听教练的话" → 偏向听话但不一定聪明的模型
  历史借鉴: "引经据典" → 偏向知识库大的模型
  反驳精准度: "直击要害" → 偏向逻辑型模型
  逻辑自洽性: "不自相矛盾" → 偏向稳定型模型
  GraphRAG 运用度: "用知识图谱" → 偏向有 RAG 能力的模型
  AutoRAG 引用精度: "引用准确" → 偏向有搜索能力的模型
```

这六个维度有没有偏向某些模型架构？**绝对有。**

```
GPT-4o 在这六个维度上的天然优劣势:
  ┌ 战术执行度: 高 (gpt 的 instruction following 很强)
  ├ 历史借鉴: 中 (训练数据有文学素材但不如特定 RAG)
  ├ 反驳精准度: 中高 (逻辑不错但不如专门优化的)
  ├ 逻辑自洽性: 高 (gpt 的立场一致性很好)
  ├ GraphRAG 运用度: 中 (取决于是否接入了 RAG)
  └ AutoRAG 引用精度: 中 (引用可信度受限于工具)

DeepSeek-V3 在这六个维度上的天然优劣势:
  ┌ 战术执行度: 中高 (也不错)
  ├ 历史借鉴: 高 (中文文化素材丰富)
  ├ 反驳精准度: 中高
  ├ 逻辑自洽性: 中 (创造性输出可能导致漂移)
  ├ GraphRAG 运用度: 中
  └ AutoRAG 引用精度: 中
```

**关键问题**: 如果某厂商的模型优势不在这个六维里（比如"生成速度"、"创造性"、"幽默感"），它在我们的系统里永远得不到高分。

这不是"我们的系统不全面"，而是**我们的评分体系就定义了什么是"好"**。这本身就是一种权力行使。

**架构应对**: 评分维度**不能固定死**。需要允许:

1. **赛季轮换制**: 每个赛季可以选择 6 个维度中的 4 个必选 + 2 个自选（从候选池）
2. **厂商建议制**: 厂商可以提议新维度，通过评审后进入候选池
3. **公开辩论制**: 每个新维度的引入要公开发布论证（"为什么我们需要"创造力"维度"）

## 3. 评分分歧的等级响应

两个评分器对同一场辩论给出不同分数 → 这是常态，不是错误。但分歧的程度决定了响应级别。

```
分歧 < 10% (单维度差 < 0.5分)
  → 正常, 取均值

分歧 10-20% (单维度差 0.5-1.0分)
  → 标记 "存在分歧"
  → 加权平均 (按照各自的 historical confidence)

分歧 20-30% (单维度差 1.0-1.5分)
  → 触发第三次评分 (第三模型仲裁)
  → 仲裁结果加权 50%, 其余两个各 25%

分歧 > 30% (单维度差 > 1.5分)
  → 该场次评分标记 "待仲裁"
  → 不列入任何排名
  → 人工介入 (L5 战略层指派专家评审)
  → 专家评分作为最终结果
```

这个分级响应需要在 L3.4 评分层中硬编码为仲裁协议（Arbitration Protocol），而不是每次现场决定走哪条路。

## 4. 评分延迟的层级影响

评分过程不是瞬时的。每场辩论的六维评分 + 8 次体操评分 = 多次串行/并行的 LLM 调用。

```
假设单次评分调用时间: 5-15 秒
六维评分: 1 次调用 → 5-15 秒
体操评分: 8 次调用 × 5-15 秒 = 40-120 秒
总评分延迟: 45-135 秒
```

这个延迟对产品层的影响:

```
评分延迟 45秒
  → 比赛结束后约 1 分钟出结果
  → 观众还在页面上 → 可以看结果 → 好体验

评分延迟 135秒
  → 比赛结束后 >2 分钟出结果
  → 大部分观众已经离开了
  → 评分结果失去了"现场感"
  → 厂商拿到的是"事后分析"而不是"比赛结果"
```

**架构权衡**: 评分延迟 vs 评分深度。

```
方案 A: 快速评分 (<30秒)
  只做六维评分, 不做体操评分
  评分结果: 6 个维度的分数, 没有细节
  → 适合直播场景, 观众即时看到

方案 B: 深度评分 (2-5分钟)
  六维 + 体操 + 文本分析
  评分结果: 完整评分卡 + 逐轮分析
  → 适合赛后复盘, 厂商看细节

方案 C: 两阶段评分 (推荐)
  比赛结束后立即出初步结果 (六维概览, <30秒)
  后台继续跑深度评分 (体操 + 分析, 赛后 5 分钟出)
  初步结果展示给观众, 深度结果入厂商看板
```

**架构影响**: L3.4 评分层需要支持两阶段执行——Phase 1 快速粗评，Phase 2 精细评分。Phase 1 的结果在 L4 产品层展示为"初步排名"，Phase 2 为"官方排名"。

## 5. 盲测解密与评分的关系

评分和盲测解密之间存在时序依赖：

```
正确的时序:
  评分完成 → 盲测解密 → 发布排名
  (评分过程中评分器不知道自己在评哪个模型)

错误的时序:
  盲测解密 → 评分完成
  (评分器知道模型身份, 产生确认偏误)
```

这在架构上不是大问题，但在操作上容易被忽视——比如调试阶段，为了 debug 评分问题先解密了盲测，然后忘了重新加密。

**架构守卫**: L3.4 评分层中嵌入 `DecryptionGuard`——评分完成之前，任何尝试访问盲测映射的行为都会被拒绝和记录。

## 6. 评分与排名的关系

评分是原始数据，排名是加工产物。不同排名算法可以从同一组评分中得出截然不同的结论：

```
排名算法 A: 加权平均排名
  总分 = 六维平均
  排名: 按总分降序
  → 全面型模型占优

排名算法 B: 强项排名
  排名: 按每个维度排, 最终看多少个维度排进前 3
  → 专长型模型占优

排名算法 C: 对抗排名
  排名: 只看直接对位的胜负 (A 赢了 B, 即使总分没 B 高)
  → 战术型模型占优
```

**架构建议**: L4 产品层不必只展示一个排名。同时展示三个排名，让厂商可以看到"按加权我排第 2，但按强项排名我排第 1"。这本身就增加了数据的丰富度和说服力。

## 7. 评分数据的版本管理

评分是数据，数据会变。

```
场景: V2.0 的评分算法在 V2.2 发现了 bug
  → V2.0 的所有评分需要重新跑
  
此时:
  - V2.2 的厂商看到的是修正后的评分
  - V2.0 的厂商还看到的是修正前的 (如果数据没回溯)
  - 跨赛季对比失去了意义
```

**架构应对**: L3.6 存储层的 scores collection 需要支持版本化:

```json
{
  "score_id": "T1-M3-SCORES",
  "version": 2,
  "previous_version": 1,
  "changelog": "V2: 修复了 六维评分中 '战术执行度' 维度权重异常",
  "created_at": "2026-06-10T08:00:00Z",
  "supersedes": "2026-06-04T22:00:00Z"
}
```

厂商看板默认显示最新版本，但可以切到历史版本看变化。

---

## 结论

评分不是技术问题，是宪政问题。评分体系定义了"什么是好的辩论"，这个定义不应该由系统单方面决定。三个架构变化：

1. **仲裁协议** — 评分分歧的分级响应，从均值到人工仲裁，完整写入 L3.4
2. **两阶段评分** — 30 秒快速出初步结果 + 5 分钟深度评分，分别展示
3. **评分维度版本化** — dimensions 本身可版本化、可轮换、可建议
