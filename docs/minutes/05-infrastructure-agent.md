# Infrastructure Agent · 生存力层级分析

> **发言时间**: SESSION-001, 第五席
> **视角**: 系统在最恶劣条件下的最小可运行状态
> **核心问题**: 如果明天所有 API 都不可用，系统还剩什么？

---

## 1. 生存力分层

"系统挂了"不是一个状态，是一系列状态，每个层级有自己的挂法：

```
层级          挂了的表现                       恢复手段
──────────────────────────────────────────────────────────────
L1 工具层     API 超时/断网/服务器 502           重启/切换/等待
L2 模块层     某个模块返回异常数据                模块重跑/跳过
L3 系统层     编排器崩溃/进程退出                 supervisor 重启
L4 产品层     UI 加载不出/数据不刷新              前端缓存兜底
L5 战略层     无 (L5 没有技术栈, 不影响在线业务)  N/A
```

**核心问题不是 "怎么不挂"，是 "挂了之后系统还有多少可用性"**。

我们把系统在不同挂法下的剩余可用性画出来：

```
全部正常运行时: 100% 可用
├─ L1 Gemini API 挂了 → 80% 可用
│   (迁移到 GPT-4o 降级, 但辩手质量下降)
├─ L1 R2 挂了 → 90% 可用
│   (本地 JSON 兜底, 只是不能同步)
├─ L2 评分模块挂了 → 70% 可用
│   (辩论还能跑, 只是不出评分)
├─ L3 编排器挂了 → 0% 可用 (最严重)
│   (需要进程管理工具自动恢复)
└─ L4 前端挂了 → 60% 可用
    (CLI 模式还能运行, 只是没 UI)
```

**架构要求**: 系统设计必须明确每个组件的生存力等级，以及对应的降级路径。

---

## 2. 降级路径的层级设计

降级不是"一键全降"，是分级降级。每一层独立决定是否降级：

```
L1 工具层降级策略:
├─ Gemini 超时 → 切换到 GPT-4o (如果可用)
├─ GPT-4o 也超时 → 切换到本地小模型 (llama.cpp)
├─ 所有模型都超时 → 等待 + 重试, 直到超时阈值耗尽
└─ 彻底断开 → 报告失败, 不降级
   (辩论引擎没有模型就没法跑, 这是硬依赖)

L2 模块层降级策略:
├─ Coach 模块超时 → 跳过 Coach, 使用默认策略
├─ Scoring 模块超时 → 标记 "评分中", 赛后补跑
├─ TTS 模块超时 → 跳过 TTS, 纯文字辩论述出
└─ Audit 模块失败 → 写入本地缓存, 恢复后补传

L3 系统层降级策略:
├─ Orchestrator 进程崩溃 → systemd/supervisor 自动拉起
├─ 拉起后检查未完成比赛 → 恢复执行
└─ 如果恢复失败 → 标记为 "异常中断", 邀请评审手动裁定

L4 产品层降级策略:
├─ 实时数据流断 → 显示缓存的上一次比赛结果
├─ 看板 API 超时 → 显示 "数据加载中" + 空状态
└─ 完全无法加载 → 显示友好的错误页 + 联系管理员按钮
```

**每个降级路径需要定义**: 触发条件、降级动作、恢复条件、数据完整性影响。

## 3. 依赖图与单点故障分析

鲲鹏志服务的核心依赖：

```
辩论系统运行
├── 必须在线 (否则不能运行):
│   ├── 至少 1 个 LLM API (Gemini/GPT/Claude/DeepSeek 任何一个)
│   ├── 本地文件系统 (写 JSON)
│   └── Python 运行时
│
├── 强烈建议在线 (否则体验下降):
│   ├── liteLLM 代理 (模型切换和负载均衡)
│   ├── Cloudflare R2 (远程备份)
│   └── Git 仓库 (同步代码)
│
├── 锦上添花 (离线不影响核心):
│   ├── TTS 服务
│   ├── GraphRAG
│   ├── 螺线谱前端
│   └── Heroku 部署
│
└── 产品层依赖 (不影响引擎):
    ├── ATP MongoDB
    ├── APEX 看板
    └── 厂商认证服务
```

**单点故障 (SPOF) 分析:**

| SPOF | 影响 | 解决方式 |
|------|------|---------|
| liteLLM 代理 | 所有模型调用失效 | 预留直接调用 LLM API 的后备路径 |
| 本地文件系统满 | 无法写入比赛数据 | 磁盘监控告警 + 自动清理策略 |
| Python 运行时损坏 | 什么都跑不了 | 容器化部署 (Docker) |
| Heroku 账号封禁 | 线上服务下线 | 辩论引擎本地可运行, 不依赖 Heroku |

## 4. 冷热数据分层

比赛数据不是平等的。有些数据需要秒级访问，有些可以存冷备。

```
热数据 (比赛进行中 / 最近 7 天):
  - 进行中的比赛状态
  - 当前锦标赛的 blind_map (加密)
  - 最近 7 天的评分结果
  存储: 本地 JSON + 内存缓存
  访问延迟: <100ms

温数据 (最近 30 天):
  - 已完成比赛的完整 episode
  - 评分卡和历史排名
  存储: 本地 JSON + R2
  访问延迟: <5s

冷数据 (超过 30 天):
  - 历史比赛全文
  - 教练策略归档
  - 赛后反思日志
  存储: R2 (本地只保留索引)
  访问延迟: <30s
```

分层策略的架构要求：
1. 数据迁移策略: 比赛结束后 T+7 自动标记为温数据
2. 数据过期策略: 冷数据保留至少一个赛季
3. 恢复策略: 从 R2 冷备恢复到本地

## 5. 性能预算层级

每个层级有性能预算，超过预算就需要扩容（或降级）：

```
辩论引擎单场性能预算:
  LLM 调用次数: 30 次 (含 Coach + Debater + Scorer)
  单场总推理时间: 8-20 分钟 (取决于模型响应速度)
  单场总输入 tokens: ~2.5M
  单场总输出 tokens: ~50K
  本地磁盘: ~500KB JSON

锦标赛 (9 场) 性能预算:
  LLM 调用次数: 270 次
  总运行时间: 72-180 分钟
  总输入 tokens: ~22.5M
  总输出 tokens: ~450K
  本地磁盘: ~4.5MB JSON
```

当锦标赛规模扩大到 50 场时，性能预算线性增长。预算超出时需要的决策:

```
预算超出 → 选项:
  ├─ 减少每场轮次 (8轮 → 4轮)
  ├─ 减少并发模型 (8个 → 4个)
  ├─ 跳过 Coach Whisper (降低 16 次调用/场)
  └─ 跳过 Teahouse (降低 1 次调用/场)
```

这些选项需要 L5 战略层授权，L3 系统层配置。

## 6. 部署拓扑的层级选择

当前架构支持三种部署拓扑，各自有生存力特征:

```
拓扑 A: 纯本地 (当前)
  辩论引擎: MBP 本地 Python 进程
  存储: 本地文件系统
  TTS: 本地 edge-tts
  LLM: 通过 liteLLM 代理 → 云端
  生存力: ★★★★☆ (只有 LLM 依赖网络)
  适用: 开发 / 单人跑锦标赛

拓扑 B: 本地引擎 + 远程存储
  辩论引擎: MBP 本地
  存储: 本地 + R2 远程备份
  TTS: 本地
  LLM: 通过 liteLLM 代理 → 云端
  生存力: ★★★☆☆ (依赖 R2 和 LLM 网络)
  适用: 多人协作, 数据共享

拓扑 C: 全远程 (Heroku + ATP)
  辩论引擎: Heroku 云端
  存储: ATP / MongoDB
  TTS: 云端 edge-tts
  LLM: 直接调用 → 云端
  生存力: ★★☆☆☆ (全链网络依赖)
  适用: 厂商看板 / 公开 demo
```

**架构要求**: 辩论引擎的核心代码不应绑定任何部署拓扑。L3.6 存储层应该支持 `LocalStore` 和 `RemoteStore` 两种实现，通过配置切换。

---

## 结论

生存力不是"不挂"，是"挂了还能剩多少"。核心架构要求：

1. **每层独立降级** — 不因为评分挂了就影响辩论引擎
2. **冷热数据分层** — 热数据本地缓存，冷数据 R2 归档
3. **部署拓扑解耦** — 核心引擎不绑定 Heroku 或本地
4. **性能预算预先定义** — 超出预算时自动触发降级选项
