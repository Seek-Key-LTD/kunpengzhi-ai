# Data Agent · 信息层级分析

> **发言时间**: SESSION-001, 第六席
> **视角**: 数据的生命周期和可见性边界
> **核心问题**: 什么数据流到哪一层就该停？谁有权看到什么？

---

## 1. 数据不是平的，是层级的

这是整个系统中最容易被低估的架构问题。大多数辩论系统（包括我们当前的设计）把数据当作"存储在某个地方的东西"。但实际上，**数据是分层的，每层有自己的可见性、沉淀速度、生命周期和权属**。

```
数据层级           内容                        可见性
────────────────────────────────────────────────────────
表层数据          比赛结果、排名、雷达图      所有人都能看到 (包括未注册用户)

中层数据          逐轮辩论文本、教练策略摘要   参赛厂商 + 注册 BYOK 评分员

深层数据          盲测映射、原始评分日志       仅管理员 + 赛后审计

底层数据          LLM 调用日志、Token 消耗     仅系统管理员 (运营层面)
```

**核心原则**: 数据往上层流动时，必须经过脱敏和聚合。底层原始数据永远不会直接暴露给上层。

---

## 2. 数据的 4 个生命周期阶段

每一条数据从产生到消亡，经历 4 个阶段，每个阶段有不同处理要求：

```
Phase 1: 生成 (辩论进行中)
  特征: 数据在产生但未确认
  状态: volatile (易失的——断网可能丢失)
  存储: 仅内存 + 临时文件
  访问: 仅当前 Orchestrator 进程
  备份: 无 (没必要备份半成品)

Phase 2: 确认 (辩论结束, 评分完成)  
  特征: 数据结构完整, 但未经审计
  状态: candidate (待审计)
  存储: 本地 JSON (热存储)
  访问: Orchestrator + L4 产品层 (只读)
  备份: 立即同步到 R2
  标记: status:"pending_audit"

Phase 3: 审计 (比赛结束 +1 小时)
  特征: 完整性校验通过, 哈希链生成
  状态: committed (已存证)
  存储: 本地 + R2
  访问: L4 产品层按角色展示
  标记: status:"committed", hash:"sha256..."

Phase 4: 归档 (赛季结束)
  特征: 不再频繁访问, 但需要保留
  状态: archived
  存储: 仅 R2 (本地可清除)
  访问: 通过 L4 产品层的历史查询 (延迟 < 30s)
  标记: status:"archived", season:"V2-SNAPSHOT"
```

## 3. 数据级别的访问控制

目前系统没有正式的访问控制层级。所有数据在本地 JSON 中可视。未来开放厂商看板时，这会成为严重问题。

建议的数据访问层级（从最开放到最限制）：

```
Level 0: 公开数据
  谁都能看, 不需要认证。
  内容: 比赛结果 (正反方得分, 总分)
  发布: 赛后自动公开, 同步到 GitHub Release
  
Level 1: 注册数据
  需要注册账号。
  内容: 六维雷达图 (按场次/按模型聚合)
  发布: 赛后可用

Level 2: 参赛方数据  
  需要是参赛厂商的认证账号。
  内容: 自己模型在所有场次的逐轮表现
  发布: 盲测解密后可用

Level 3: 管理员数据
  需要管理员权限。
  内容: 全局盲测映射 (历史)
  发布: 仅运营需要时

Level 4: 审计数据
  需要超级管理员权限 + 审计理由。
  内容: LLM 原始调用日志、Token 消耗明细
  发布: 仅在争议时
```

**架构要求**: L3.6 存储层的每条数据记录需要有 `access_level` 字段。L4 产品层根据用户角色过滤数据。

## 4. 数据一致性的层级保证

当数据跨层（本地 + R2 + 未来 ATP）存在时，一致性成问题。

```
场景: 比赛结束后:
  1. Episode 写入本地 JSON (成功)
  2. 同步到 R2 (网络波动, 失败)
  3. 本地 JSON 被标记为 "已同步"

此时, 如果本地磁盘损坏, 数据丢失。
```

**一致性协议**: 简单的主从模型 (Leader-Follower)

```
Leader: 本地 JSON (权威数据源, ground truth)
Follower: R2 (异步复制, 可以落后)

写入流程:
  Write → Local JSON (sync) → 
    ├─ 成功: 标记 status:"synced" (但 R2 可能还没收到)
    └─ 失败: 标记 status:"failed"

同步流程:
  Local → 检查未同步数据
  Local → 推送到 R2
  R2 确认 → Local 标记 status:"backed_up"

恢复流程:
  Local 损坏 → 从 R2 拉取 → 重建 Local
  R2 不可用 → 继续用 Local → 恢复后补传
```

## 5. 图数据与文档数据的层级关系

当前系统涉及两种数据范式，它们应该存在于不同层级：

```
文档数据 (Document Store):
  内容: Episode, Scores, CoachStrategy
  特点: 结构化 JSON, 按 ID 查询
  层级: L3.6 主存储
  格式: JSON 文件 / MongoDB Document
  
图数据 (Graph):
  内容: 角色关系 (打斗/联盟/师徒)、全局谱系
  特点: 实体 + 关系, 需要跨文档查询
  层级: L3.6 辅助存储
  格式: GraphRAG / NetworkX
```

**架构选择**: 文档数据是"源", 图数据是"派生产物"。

```
源数据: 每场比赛的 Episode JSON
  → 提取: 角色出现频率、对话关系、引用的书中段落
  → 构建: 角色关系图谱
  → 存储: GraphRAG

这就要求:
  - 图数据不是手动维护的, 而是从文档数据自动派生的
  - 图数据的 schema 不能反作用于文档数据
  - GraphRAG 的构建是一个 **异步的 ETL 管道**
```

这个 ETL 管道本身也是烧 token 的点——每次从 episode 中提取角色关系都需要 LLM 推理。

## 6. 数据垃圾的层级清除

数据会积累，不是所有数据都值得保留。

```
需要自动清除的数据:
  1. 调试日志: 保留 7 天
  2. 半成品比赛 (status:"aborted"): 保留 30 天后标记 "可用存档"
  3. 比赛进行中的临时 checkpoint: 比赛结束后立即清除
  4. LLM 原始调用日志: 保留 90 天 (合规需要)
  5. BYOK 评分员的 API Key: 比赛结束后立即从内存清除

不需要清除的数据:
  1. 已完成比赛的 Episode: 永久保留 (这是核心资产)
  2. 评分卡: 永久保留
  3. 锦标赛配置: 永久保留 (用于复现)
  4. 盲测解密记录: 永久保留 (审计需要)
```

**架构要求**: L3.6 存储层需要有 TTL (Time-To-Live) 字段，L2 模块层有数据清理守护进程。

## 7. 一场比赛的数据流（完整链路）

这是从数据视角看一场比赛的完整生命周期，标注每个数据在哪一层生成、静止、消亡：

```
1. L4 产品层: 管理员创建赛事 → 写入 Tournament 配置
   ↓ 数据下沉到 L3.6

2. L3.1 编排层: 生成 MatchPlan (含 blind_map)
   ↓ blind_map 加密后存入 L3.6 (access_level:3)
   ↓ MatchPlan (无 blind_map) 推送到 L3.2

3. L3.2 辩论引擎: 开始逐轮辩论
   ↓ 每轮输出写入 L3.6 (status:"volatile")
   ↓ 但只保留最后一轮的完整上下文(轮次间不断追加)

4. L3.3 教练层: Pre-debate + Round Whisper
   ↓ 策略写入 L3.6 (access_level:2, 赛后对参赛方可见)

5. L3.2 辩论引擎: 辩论结束
   ↓ 完整 Episode 写入 L3.6 (status:"candidate")
   ↓ 同步 R2 触发器启动

6. L3.4 评分层: 评分执行
   ↓ 评分卡写入 L3.6 (access_level:1, 公开)

7. L3.3 教练层: 赛后反思 (Cycle 2 of Coaching)
   ↓ coach_lessons 写入 L3.6 (access_level:3, 仅管理员)
   ↓ 这是唯一不向参赛方公开的教练产物

8. L3.1 编排层: 审计
   ↓ 哈希链生成 → 写入 L3.6 的 audit 子集
   ↓ 哈希值推送到 GitHub Release (公开)

9. L4 产品层: 解密触发
   ↓ L5 战略层决策解密时间 → L4 执行解密
   ↓ blind_map 解密 → 数据从 access_level:3 升级到 access_level:2
   ↓ 参赛方看到自己模型的身份映射

10. L3.6 存储层: 状态更新
    ↓ Episode 标记为 status:"committed"
    ↓ 7 天后移入温存储
    ↓ 赛季结束后移入冷存储
```

这个流中的每个箭头都是一个接口契约——定义了数据结构、访问权限、状态转换。

---

## 结论

数据不是被动的存储物，它是系统的神经系统。三个架构要求：

1. **数据生命周期 4 阶段** — 生成 → 确认 → 审计 → 归档，每阶段有不同处理策略
2. **5 级访问控制** — Level 0-4，每个数据记录必须标注访问级别
3. **文档数据是源，图数据是派生** — GraphRAG 是 ETL 管道的产物，不是源数据
