## 1. Security Agent 开篇：身份层级与盲测保密

### What-If 1: 比赛中途盲测映射被泄露

**场景**: 某参赛厂商通过侧信道（tokens per second 的差异、回复格式的规律）辨认出对手的模型身份，并在辩论中针对性攻击。

**响应分析 (Security Agent):**

```
风险等级: 🔴 HIGH
影响范围: L4 产品层 (信任危机) → L3 评分层 (评分失效) → L5 战略层 (邀约可信度崩盘)
```

**防御层级建议:**

| 层 | 措施 | 谁执行 |
|----|------|--------|
| L1 工具层 | 所有模型统一输出格式（后处理去格式差异） | Infrastructure Agent |
| L2 模块层 | Token 延迟随机化（消除响应速度侧信道） | Debate Engine Lead |
| L3 系统层 | 每次推理随机偏移输出 buffer（消除 token 模式指纹） | Scoring Lead |
| L4 产品层 | 赛后解密 + 公开盲测报告 | 我（Manager） |
| L5 战略层 | 参赛协议中写入：测出对方身份但不报告 = 违规 | 你 |

**追加疑问**: 如果模型本身在生成时植入隐水印（如特定位置的标点模式）？
→ L2: 所有输出通过去水印过滤器再展示

### What-If 2: 匿名参赛的厂商被曝出身份

**场景**: "匿名参赛"的厂商被发现，造成竞争对手不满。

**响应:** 需要一个 **「匿名等级」** 概念:
- L1 匿名: 赛后也不公布（完全隐姓埋名）
- L2 匿名: 赛后公布模型类型但不公布具体版本
- L3 匿名: 赛后公布是 "中国某厂商" 但不具体

**架构调整**: L2 模块层增加 `anonymity_level` 字段贯穿管道

## 2. Product Agent 发问：产品层级断裂

### What-If 3: 观众看不懂评分雷达图

**场景**: V2.1 放出厂商看板，但第一批外部用户反馈 "这六个维度是什么意思？"

**响应分析 (Product Agent):**

```
风险等级: 🟡 MEDIUM
影响范围: L4 产品层 (用户体验) → L5 战略层 (邀约体面感)
```

**决策树:**

```
观众反馈 "看不懂"
├─ 是 <5% 的用户 → 忽略, 加 tooltip
├─ 是 5-30% 的用户 → 加教育弹窗 + 1:30 教程视频
├─ 是 30-60% 的用户 → 简化到 3 个聚合维度
└─ 是 >60% 的用户 → 退回 L5 重新定义产品价值主张
```

**架构调整**: L4 产品层增加「聚合维度」概念 (默认展开6维，可选折叠为3维)

### What-If 4: 某大厂要求定制辩论格式

**场景**: 腾讯来问 "你们能支持 3v3 吗？我们三个模型想一批上"

**响应:**

```
L5 战略层决策:
├─ 拒绝定制 (保持竞技公平) → 可能流失大客户
├─ 做定制 (腾讯VIP格式) → 维护成本 + 其他厂商也要:
│   ├─ 做一次性的 → 成本高
│   └─ 做成可配置的 → 架构灵活但复杂度上升
└─ 开放辩论格式配置: 2v2/3v3/4v4 可配置
```

**推荐路径**: 辩论引擎层 (L3.2) 从设计时就支持可配置席位数量，把 `num_seats=8` 做成参数而不是硬编码。这样 3v3 就是改个配置的事。不做硬编码的定制。

### What-If 5: "讲茶大堂"评论出现极端内容

**场景**: AI 场外评论员生成争议性内容（政治敏感、种族歧视），被截图传播。

**响应**: 
- L4: 讲茶大堂的 AI 评论需要经过**内容过滤器**再展示
- L3: 增加 `core/content_filter.py` 模块
- L5: 准备危机公关预案

## 3. Debate Coach Agent 挑战：辩论引擎层

### What-If 6: 教练策略在关键时刻自相矛盾

**场景**: Pre-debate Coach 说 "攻击对方的逻辑漏洞"，但 Round Coach Whisper 说 "防守为主，稳住观点"。辩手（AI）不知道该听哪个。

**响应分析 (Coach Agent):**

```
风险等级: 🔴 HIGH
影响范围: L3.3 教练层 (逻辑不一致) → L3.4 评分层 (战术遵从度无法评估)
```

**根因**: Pre-debate 策略和 Round Whisper 是两次独立调用，没有一致性检查。

**决策树:**

```
两条指令矛盾
├─ 训练时间允许 → 在 Coach 输出时增加 consistency_check
│   ├─ 让 LLM 自己检查两段策略的逻辑矛盾
│   └─ 如果检测到矛盾 → 拒绝 Round Whisper, 复用 Pre-debate
├─ 训练时间不允许 → 强制 Round Whisper 引用 Pre-debate 策略:
│   └─ Whisper prompt 加入: "你的建议不可与 pre-debate 策略矛盾"
└─ MVP 阶段 → 信任 LLM, 标记矛盾到赛后分析
```

**架构调整**: L3.3 增加 `ConsistencyValidator` 模块

### What-If 7: 同一模型的左右互搏

**场景**: 正方用 DeepSeek-V3, 反方也用 DeepSeek-V3（不同实例）。正方教练给 DeepSeek 的策略被反方的 DeepSeek 反向推理出来（"如果我是正方教练我会建议什么"），形成诡异的对称辩论。

**响应**: 
- 不算 bug，但辩题质量下降
- L5: 可以把它包装成 feature——"镜像对决"
- L3: 赛后分析标记模型重复，但不禁止

### What-If 8: 8 轮辩论有一轮卡住

**场景**: Round 4 自由辩论阶段，某模型的 API 调用超时（超过设定的 30s timeout）。

**响应:**

```
辩论中断 (Round N)
├─ 重试策略:
│   ├─ 重试 1 次 → 成功 → 继续
│   └─ 重试 1 次 → 仍超时
│       ├─ 跳过该轮 → 标记 "skipped"
│       └─ 用备用模型替代 → 但不是原定盲测对手
├─ 熔断策略:
│   └─ 连续 3 次超时 → 整场标记 "aborted", 不记入锦标赛
└─ MVP: 简单跳过, 赛后说明 "该场为 incomplete"
```

**架构调整**: L3.2 辩论引擎层增加 `retry_policy` 和 `circuit_breaker` 配置。

## 4. Scoring Agent 补刀：评分层

### What-If 9: 六维评分严重偏向某一方

**场景**: 同一场辩论，用 Gemini 2.5 Flash 评分 → 正方赢 4.2 vs 反方 3.8。用 GPT-4o 评分 → 反方赢 4.0 vs 正方 3.5。

**响应分析 (Scoring Agent):**

```
风险等级: 🟡 MEDIUM
影响范围: L3.4 评分层 (可信度) → L4 产品层 (厂商信任)
```

**决策树:**

```
评分不一致
├─ 差异 < 15% → 正常, 取平均
├─ 差异 15-30% → 标记 "争议场", 需要人工仲裁
└─ 差异 > 30% → 评分器失效, 退回 L5:
    ├─ 重新定义评分维度
    ├─ 增加第三方评分器
    └─ 放弃 LLM 自动评分, 全量 BYOK
```

**架构调整**: L3.4 增加 `scorer_ensemble` 概念——至少 2 个模型评分，差异太大时触发人工仲裁。

### What-If 10: BYOK 评分员被操纵

**场景**: 某厂商用 100 个 API Key 刷分，全部给自己的模型打 5 分。

**响应:**

```
BYOK 水军检测:
├─ L1 检测: 同一 IP 段的多个 key → 阈值告警
├─ L2 检测: 评分向量的余弦相似度:
│   ├─ 同一集群的评分相似度 > 0.9 → 标记可疑
│   └─ 人工复查
└─ L3 检测: 评分偏离历史均值 > 2σ → 降权处理
```

**架构调整**: L3.4 评分层增加 BYOK 评分质量检测模块，包含聚类检测 + 异常值检测。

## 5. Data Agent 收网：数据层

### What-If 11: 数据 schema 需要向后兼容

**场景**: V2.0 定稿了 ATP Schema，但 V2.1 要加一个字段（比如 `audience_votes`），V2.0 的数据怎么办？

**响应**: 
- 不是问题——JSON Schema 天生支持 optional fields
- 但需要 L3.6 存储层提供 schema migration 文档
- 每个版本的数据 + 一个 `schema_version: "v2.0"` 字段

### What-If 12: 本地 JSON 和 R2 数据不一致

**场景**: 锦标赛执行到一半断网。本地 JSON 写了，但 R2 同步失败。

**响应**: 
- L3.6 存储层设计为 **local-first, R2-as-buffer**:
  - 写操作先写本地
  - 异步同步到 R2
  - 下次启动时检测未同步数据 → 补传
- L2 模块层: `SyncManager` 模块

## 6. Infrastructure Agent 兜底：工具层

### What-If 13: 所有 GCP credit 被烧光，API 全部 429

**场景**: Pipeline 并发跑太狠，GCP 配额被打满，所有调用来不及返回。

**响应**:

```
API 配额打满
├─ L1: 立即熔断, 暂停所有并发
├─ L2: 降级到低开销模式:
│   ├─ 只做文字辩论, 不做 Coach Whisper
│   ├─ 减少 Teahouse 调用
│   └─ 只做体操评分, 不做六维评分
├─ L3: 降级到最小可运行配置:
│   └─ 只跑辩论, 不评分, 不入库
└─ L4: 通知管理员, 等待配额恢复
```

### What-If 14: Heroku 重部署或宕机

**场景**: 上线前 Heroku 出现问题，需要切到其他平台。

**决策**: L5 层决定是否要预留备选部署方案。
**架构调整**: 辩论引擎本身与部署解耦——Chainlit 只是 UI 层。核心引擎是 CLI 可运行的，不依赖 Heroku。

## 7. TTS Agent 彩蛋：感知层

### What-If 15: TTS 生成的语音被识别为真人声音

**场景**: Omni-TTS 过于逼真，观众以为是真的八仙在说话，产生伦理质疑。

**响应**: 
- L4: 每条 TTS 输出添加隐形水印（特定频段标记）
- L4: UI 明确标注 "AI 合成语音"

### What-If 16: 八席声纹互相串扰

**场景**: 因为 seed 或 prompt_template 接近，两个角色的 TTS 声音听不出来区别。

**响应**:
- L2 TTS 模块: 入场前做声纹相似度测试
- 相似度 > 0.8 → 调整 seed 或 prompt_template 直到 < 0.5

## 8. Manager 总结：跨层级级联崩溃分析

这是最关键的——**不是一层出事，是一件事引发连锁反应**。

### 级联场景 A: API 超时 → 多米诺

```
L1 工具层: Gemini API 超时 (30 秒)
  → L2 辩论模块: 重试 1 次, 再次超时
    → L3.2 辩论引擎: 跳过该轮, 标记 "skipped"
      → L3.4 评分层: 收到了 incomplete 的 episode
        → 评分不知道该跳过还是补全
          → L3.1 编排层: 无法判断该 episode 是否算数
            → L4 产品层: 厂商看板出现一场有缺失的比赛
              → L5 战略层: 需要决定 "算不算锦标赛记录"
```

**架构落地**: 在 L3.1 编排层增加明确的 `EpisodeValidityRule`:

| 条件 | 判定 | 
|------|------|
| 8 轮完整跑完 | valid |
| 1-2 轮被跳过 | partial — 可计分但标注 |
| >2 轮被跳过 | invalid — 不计入锦标赛 |

### 级联场景 B: BYOK 评分争议 → 媒体危机

```
L2 BYOK 模块: 某厂商被 AI 检测为刷分
  → L3.4 评分层: 降权处理
    → 刷分厂商不满, 发社交媒体质疑
      → L4 产品层: 需要公关回应
        → L5 战略层: 
          ├─ 公开 BYOK 检测算法 → 帮助水军绕过
          └─ 不公开 → 被质疑黑箱操作
```

**架构落地**: BYOK 检测策略需要 L5 定调——**公开检测逻辑但不公开阈值**。即 "我们检测评分相似度"（公开）但 "相似度多高算可疑"（不公开）。

### 级联场景 C: 第一场锦标赛就出 bug

```
L2 辩论模块: 某个模型返回了空字符串
  → L3.2 辩论引擎: 空字符串入库
    → L3.4 评分层: 评分器输入了空文本, 产生垃圾评分
      → L3.6 存储层: 垃圾数据入库
        → L4 产品层: 第一场锦标赛的报告有脏数据
```

**架构落地**: 每层增加 **输入校验守卫** (Guard):

| 层 | 守卫 | 检查 |
|----|------|------|
| L3.2 辩论引擎 | `SpeechValidator` | 发言不为空、不超长、有实际内容 |
| L3.4 评分层 | `ScoreSanityCheck` | 分数在 1-5 范围, 不全是极端值 |
| L3.6 存储层 | `IntegrityCheck` | Episode 结构完整, 必填字段不缺 |

## 总结：从 What-If 得出的架构加强

| 发现 | 影响层级 | 架构动作 |
|------|---------|---------|
| 教练策略可能自相矛盾 | L3.3 | 增加 `ConsistencyValidator` |
| LLM 评分器有偏见 | L3.4 | `scorer_ensemble` 多模型评分 |
| BYOK 可被刷分 | L3.4 | 聚类检测 + 异常值降权 |
| 盲测有侧信道泄露 | L2 | 统一输出格式 + Token 延迟随机化 |
| 单轮超时可级联 | L3.1/L3.2 | `retry_policy` + `EpisodeValidityRule` |
| 空数据入库破坏分析 | 跨层 | 每层加 `Guard` 输入校验 |
| TTS 声纹相似度 | L2 | 前置相似度测试 |
| 匿名有等级 | L4 | 三级匿名制 |
| 辩论格式不可硬编码 | L3.2 | 席位数为配置参数 |
| 评分争议需要仲裁 | L3.4/L4 | 明确降级路径 + 人工介入点 |
