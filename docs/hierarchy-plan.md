# 🦅 鲲鹏志 V2.0 · 层级架构蓝图

> ## 单一任务
> **设计并输出鲲鹏志 V2.0 的完整层级架构 (Hierarchy Design)**。
>
> 不是写代码。不是画 API 路由。不是写 SQL DDL。
> 而是定义：**"这个系统从顶到底、从外到内、从战略到执行的每一层是什么、谁负责、怎么衔接。"**
>
> 目标 Credit 烧录: **5000 credits** (脑力密集型架构设计工作量)
> 交付时间: 明天早上 08:00
> 验收方式: 和你通读，逐层确认

## 这个文档是给谁用的？

这个层级架构设计一旦产出，会有三类读者：

| 读者 | 用它做什么 |
|------|----------|
| **你（产品拥有者）** | 拿着这个去跟大厂聊"我们的系统层级架构"，证明工程成熟度 |
| **10+ 即将加入的 agents** | 读完知道自己属于哪个层级、和谁对接、决策权到哪 |
| **我（Manager）** | 基于这个做排期、分派任务、判定什么决策在哪层解决 |

**所以这不是一个技术文档，这是系统的宪法。**

## 我要设计的 5 层 Hierarchy

```
┌──────────────┐
│  L5 战略层   │  ← 你（产品拥有者）
├──────────────┤
│  L4 产品层   │  ← 我（Manager / 架构师）
├──────────────┤
│  L3 系统层   │  ← 各 Lead Agents
├──────────────┤
│  L2 模块层   │  ← 各 Developer Agents
├──────────────┤
│  L1 工具层   │  ← LLM / R2 / GCP / APEX / ATP
└──────────────┘
```

### 每层要涵盖的设计维度

| 维度 | 说明 |
|------|------|
| **职责边界** | 这一层管什么、不管什么 |
| **输入/输出契约** | 这一层接收什么、产出什么 |
| **决策权** | 哪些决策在这一层做、哪些要上报 |
| **失败模式** | 这一层如果崩了会发生什么、怎么恢复 |
| **与上下层的接口** | 向上汇报什么、向下指挥什么 |
| **人员配属** | 哪类 agent 驻守这一层 |

## 要产出的具体内容（按层级）

### L5 · 战略层 — The Why

**决策者**: 你（产品拥有者）
**设计工作量**: 500 credits

设计内容:
1. **愿景层级分解**: "鲲鹏志"这个品牌的三层含义
   - 核心层: 辩论引擎（技术护城河）
   - 应用层: 联赛/评测/跨文化（产品形态）
   - 辐射层: 大厂生态 / 学术价值 / 文化输出（影响力）
2. **大厂邀约的价值链**:
   - 大厂为什么要来？→ 针对 4 种 Persona 的价值主张层级
   - 邀约 Kit 的信息层级: 5 秒电梯演讲 → 5 分钟 demo → 5 小时技术对接
3. **里程碑决策树**: 什么条件下要做出分支决策（例如：V2.0 跑通后是先发邀约还是继续开发？）

### L4 · 产品层 — The What

**决策者**: 我（Manager / 架构师）
**设计工作量**: 1500 credits

设计内容:
1. **产品层级分解**: 从用户角度看到的系统
   - **观赛层**: 用户看到什么？(辩论流 / 螺线谱 / 评分雷达 / 讲茶大堂)
   - **参与层**: 厂商能做什么？(上传模型 / 查战绩 / 看数据)
   - **治理层**: 主办怎么做？(排期 / 选辩题 / 裁决争议)
2. **赛事层级体系**:
   - 一场辩论的完整层级: 锦标赛 → 赛区 → 场次 → 对宫辩论 → 轮次
   - 每层的身份概念:
     - 竞技场 (Arena): 连场辩论构成一个竞技场
     - 锦标赛 (Tournament): 多场构成锦标赛
     - 赛季 (Season): 多锦标赛构成赛季
   - 每个层级的计分规则、排名算法、历届数据
3. **用户角色层级**:
   ```mermaid
   graph TD
     A[全员] --> B[观众]
     A --> C[参赛厂商]
     A --> D[主办方]
     B --> B1[匿名观众]
     B --> B2[注册用户/BYOK评分员]
     B --> B3[VIP赞助商]
     C --> C1[普通参赛]
     C --> C2[盲测参赛]
     C --> C3[匿名参赛]
     D --> D1[评审员]
     D --> D2[管理员]
     D --> D3[超级管理员]
   ```
4. **信息层级**: 哪些数据在哪个身份层级可见

### L3 · 系统层 — The How (Architecture)

**决策者**: 各 Lead Agents（我定义框架）
**设计工作量**: 1500 credits

设计内容:
1. **系统层级分解**: 从技术角度看到的系统
   ```
   L5 战略 → L4 产品 → L3.1 编排层 (Orchestration)
                        ├── L3.2 辩论引擎层 (Debate Engine)
                        ├── L3.3 教练层 (Coach)
                        ├── L3.4 评分层 (Scoring)
                        ├── L3.5 感知层 (TTS/螺线谱)
                        └── L3.6 存储层 (Data Store)
                    → L2 模块 → L1 工具
   ```
2. **每层的职责合同 (Contract)**:
   - 辩论引擎层接收什么（TopicConfig + ModelPool）→ 产出什么 (Episode)
   - 教练层接收什么 (Topic + History) → 产出什么 (StrategyDoc)
   - 评分层接收什么 (Episode + Strategies) → 产出什么 (ScoreCard)
   - 存储层接收什么 (数据对象) → 产出什么 (持久化 + 可查询)
3. **每层的决策权边界**:
   - 辩论引擎层: 可以决定用哪个 prompt template，但不能决定换模型
   - 教练层: 可以决定策略的 generation params，但不能改辩题
   - 评分层: 可以决定评分权重和算法，但不能改评分维度
4. **每层的失败模式**:
   - 某层超时 → 怎么降级？
   - 某层结果异常 → 谁有权重试 / 跳过 / 人工介入？

### L2 · 模块层 — The Modules

**决策者**: Developer Agents（我定义边界）
**设计工作量**: 1000 credits

设计内容:
1. **模块分解**: 每个 L3 层的内部模块结构
   - 辩论引擎层: 开篇立论模块 / 驳论模块 / 自由辩论模块 / 总结陈词模块
   - 评分层: 六维雷达评分模块 / 体操评分模块 / 盲测核查模块
   - 教练层: Pre-debate Coach / Round Whisper Coach
2. **模块间通信协议**:
   - 同步 vs 异步调用
   - 超时阈值
   - 重试策略
   - 熔断条件
3. **模块测试层级**:
   - 单元测试（模块内）
   - 集成测试（模块间）
   - 端到端测试（跨 L3 层）

### L1 · 工具层 — The Tools

**决策者**: Infrastructure Agent（我定义选择标准）
**设计工作量**: 500 credits

设计内容:
1. **工具选择层级**: 什么层级用什么工具
   - 辩论引擎: Gemini 2.5 Flash → 什么时候升级到 Pro
   - 存储: 本地 JSON → 什么时候迁到 MongoDB / ATP
   - 部署: Heroku → 什么时候迁到更高可用性方案
2. **工具选型决策矩阵**: 每个工具选或不选的 Trade-off
3. **迁移路径**: 从当前工具到目标工具的分阶段过渡

## 每一层的输出格式

每一层设计都要输出 **3 个文档级别的产物**:

### 1. 层级定义卡 (Level Definition Card)

```
层级: L3.2 辩论引擎层
所属系统层: L3 系统层
负责人角色: Debate Engine Lead Agent
上报给: L4 产品层 (我)
调度: L2 模块层 (Developer Agents)

职责声明:
  辩论引擎层负责接收一个赛前配置 (Topic + ModelPool + RuleSet),
  执行完整的 4v4 八股辩论流程, 产出 Episode (完整对话记录)。

不负责:
  - 生成教练策略 (那是 L3.3 教练层的事)
  - 评分 (那是 L3.4 评分层的事)
  - 部署和运维 (那是基础设施层的事)

输入契约:
  - TournamentConfig (Tournament Orchestrator 下发)
  - StrategyDoc (Coach Layer 异步注入)
  
输出契约:
  - Episode (包含 8 轮 16 段发言 + 盲测映射)
  - 状态: pending → debating → waiting_coach → debating → ... → completed

失败模式:
  - 某模型超时 → 跳过该轮, episode 标注 "partial"
  - 连续 3 模型超时 → 熔断, 整场标记 "aborted", 上报 L4

决策权:
  - ✅ 决定单轮辩论的 prompt 构造
  - ✅ 决定 model generation params (temperature, max_tokens)
  - ❌ 更换模型池中的模型 (L4 的决策)
  - ❌ 修改辩题 (L5 的决策)
```

## 执行计划

### Phase 1 · L5 战略层设计 (1h · 500 credits)

1. 访谈分析: 从你的视角提炼"鲲鹏志"的三层品牌价值
2. 大厂 Persona 价值层级树: 每个 Persona 从表层到深层的价值主张
3. 邀约 Kit 的信息层级结构: 什么时间说什么
4. 里程碑决策树: 关键节点 + 分支条件

**产出**: `L5-strategy.md`
**验证**: 你读完后说"对，这就是我想的"

### Phase 2 · L4 产品层设计 (3h · 1500 credits)

1. 产品层级分解: 观赛/参与/治理三层的详细设计
2. 赛事层级体系: 竞技场 → 锦标赛 → 赛季 → 联赛的完整定义
3. 用户角色层级: 从匿名观众到超级管理员的所有角色
4. 信息可见性层级: 什么数据给什么人看

**产出**: `L4-product.md`
**验证**: 可以拿着这个去给厂商解释"你们能看到什么数据"

### Phase 3 · L3 系统层设计 (3h · 1500 credits)

1. 6 个子系统的职责卡: 辩论引擎 / 教练 / 评分 / 感知 / 存储 / 编排
2. 每层的输入输出契约: 精确到数据结构和失败模式
3. 层级间通信协议: 同步/异步、超时、重试、熔断
4. 决策权边界: 每层可以做什么、不可以做什么

**产出**: `L3-system.md`
**验证**: 10+ agents 来了后, 每人读完找到自己的层级和职责

### Phase 4 · L2 + L1 设计 (2h · 1500 credits)

1. 模块分解: 每个 L3 层的内部模块切分
2. 模块间通信协议: 每层内部模块的协作关系
3. 工具选型矩阵: 当前用啥 → 未来用啥 → 什么时候迁
4. Agent 配属表: 每个层级驻守哪些 agent、每个 agent 的汇报线

**产出**: `L2-modules.md` + `L1-tools.md`
**验证**: 明确哪个 agent 干哪块活
